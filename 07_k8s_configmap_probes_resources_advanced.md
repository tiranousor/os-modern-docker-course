# Лабораторная работа: продвинутые сценарии работы с ConfigMap, probes и ресурсами в Kubernetes

## Цель работы

Закрепить навыки эксплуатации Kubernetes-приложения и научиться диагностировать типовые проблемы, связанные с:

- обновлением конфигурации через **ConfigMap**;
- различием между **readiness**, **liveness** и **startupProbe**;
- перезапусками контейнера;
- ограничениями по **CPU** и **memory**;
- состояниями **NotReady**, **CrashLoopBackOff**, **OOMKilled** и **Pending**.

## Правила выполнения

1. Все действия выполняются в отдельном namespace: **`lab-k8s-username`**.
2. Можно использовать Deployment и ConfigMap из прошлой лабораторной работы как основу.
3. Для каждого кейса в отчете обязательно указать:
   - что было изменено;
   - какой симптом появился;
   - какими командами это было обнаружено;
   - в чем причина;
   - как проблема была исправлена.
4. В отчете должны быть:
   - YAML-файлы;
   - ключевые команды;
   - фрагменты вывода `kubectl get`, `kubectl describe`, `kubectl logs`, `kubectl get events`;
   - краткие пояснения по каждому кейсу.

## Шпаргалка команд

### Общие команды

- `kubectl get pods -n <ns>`
- `kubectl get deploy -n <ns>`
- `kubectl get svc -n <ns>`
- `kubectl get cm -n <ns>`
- `kubectl get events -n <ns> --sort-by=.metadata.creationTimestamp`

### Диагностика

- `kubectl describe pod <pod-name> -n <ns>`
- `kubectl describe deploy <deploy-name> -n <ns>`
- `kubectl logs <pod-name> -n <ns>`
- `kubectl logs <pod-name> -n <ns> --previous`

### Применение изменений

- `kubectl apply -f <file> -n <ns>`
- `kubectl delete -f <file> -n <ns>`
- `kubectl rollout restart deploy/<deploy-name> -n <ns>`
- `kubectl rollout status deploy/<deploy-name> -n <ns>`

### Доступ к приложению

- `kubectl port-forward svc/<service-name> 8080:80 -n <ns>`

## Часть 2. Startup probe и медленный старт приложения

### Задание 2. Моделирование долгого старта

Создайте отдельный Deployment, в котором контейнер стартует с искусственной задержкой.

**Идея кейса:**

контейнер существует, но приложение внутри начинает отвечать не сразу.

### Требуется

1. Создать Deployment с задержкой запуска.
2. Сначала настроить только `readinessProbe` и `livenessProbe`.
3. Наблюдать поведение pod’ика.
4. Затем добавить `startupProbe`.
5. Сравнить результат до и после добавления `startupProbe`.

### Что нужно показать в отчете

- YAML Deployment без `startupProbe`;
- симптомы проблемного старта;
- YAML после добавления `startupProbe`;
- чем изменилось поведение pod’а;
- вывод `describe pod`.

## Часть 3. Ошибка liveness probe и перезапуски контейнера

### Задание 3. Искусственно сломанная liveness probe

Возьмите рабочий Deployment и настройте в нем **ошибочную `livenessProbe`**.

Например:

- указать неверный путь;
- указать неправильный порт;
- задать слишком агрессивные параметры проверки.

### Требуется

1. Применить изменения.
2. Наблюдать, что контейнер начинает перезапускаться.
3. Зафиксировать:
   - количество рестартов;
   - события;
   - состояние pod’а;
   - описание pod’а.
4. Исправить `livenessProbe` и восстановить стабильную работу.

### Что нужно показать в отчете

- YAML с ошибочной `livenessProbe`;
- признаки проблемы:
  - `RESTARTS`;
  - возможное состояние `CrashLoopBackOff`;
  - события;
- исправленный YAML;
- финальное рабочее состояние.

3. Наблюдать завершение контейнера.
4. Найти в диагностике признак `OOMKilled`.

### Что нужно показать в отчете

- YAML с проблемной конфигурацией;
- состояние pod’а;
- вывод `describe pod`;
- где именно видно `OOMKilled`;
- исправленный вариант конфигурации.

## Часть 5. Слишком низкий CPU limit и деградация работы

### Задание 5. Ограничение CPU

Создайте Deployment с очень низким `cpu limit`.

### Требуется

1. Поднять приложение с небольшим `cpu limit`.
2. Сгенерировать нагрузку на контейнер.
3. Сравнить работу контейнера:
   - до ограничения CPU;
   - после ограничения CPU.
4. Зафиксировать, что приложение не обязательно падает, но начинает работать медленнее.

### Допустимые варианты демонстрации

- измерить время ответа;
- зафиксировать, что приложение начинает заметно тормозить;
- привести пояснение по поведению контейнера под CPU-throttling.

### Что нужно показать в отчете

- YAML с CPU-ограничением;
- способ создания нагрузки;
- симптомы деградации;
- объяснение, почему контейнер не падает, а «тормозит».
