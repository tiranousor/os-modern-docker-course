# Лекция: Оркестрация контейнеров, Docker Swarm и Kubernetes – база для практики

## Зачем вообще нужна оркестрация

При использовании Docker по стандартному сценарию обычно выполняются следующие шаги:

- собирается образ;
- запускается контейнер командой вида `docker run ...`;
- получается один экземпляр приложения на одном хосте.

Такой подход работает, пока:

- пользователей мало;
- контейнеров немного;
- один сервер (docker-хост) справляется с нагрузкой.

В реальной эксплуатации быстро возникают типовые проблемы:

### Нагрузка растет
Один контейнер или одна VM перестают справляться. Требуется запускать дополнительные экземпляры приложения.

### Нужно масштабирование
Если дополнительные контейнеры запускаются вручную, вы фактически выполняете роль «ручного автоскейлера»: отслеживаете нагрузку и сами запускаете/останавливаете копии.

### Сбои контейнеров
Контейнер может упасть – это нужно обнаружить и перезапустить.

### Сбои хоста
Если падает docker-хост (железо/VM/сеть), падает все, что на нем запущено.

### Мониторинг, балансировка, сеть, конфигурации
Чтобы поддерживать систему вручную, понадобятся мониторинг, балансировка запросов, маршрутизация, конфигурации и секреты, а также люди, которые это обслуживают.

Когда контейнеров становится десятки/сотни/тысячи, ручное управление перестает быть приемлемым. В этом случае применяются **системы оркестрации**: они автоматизируют размещение контейнеров по хостам, восстановление после сбоев, масштабирование, обновления и сетевую связанность.

## 1) Docker Swarm: что это и как он устроен

### 1.1) Swarm-кластер

Docker Swarm объединяет несколько Docker-хостов в единый кластер:

- **Manager (менеджер)** – управляющий узел: хранит состояние и принимает команды.
- **Worker (воркер)** – рабочий узел: запускает контейнеры.

Базовый сценарий создания:

1. Установите Docker на несколько машин.
2. Выберите узел, который будет manager.
3. На manager выполните `docker swarm init`.
4. Получите команду `join` из вывода.
5. Выполните `join` на воркерах – воркеры станут узлами кластера.

### 1.2) Почему «docker run на каждой ноде» – плохой план

Если масштабировать приложение так, чтобы заходить на каждую машину и выполнять `docker run`, вы столкнетесь со следующими проблемами:

- необходимость организовать доступ (учетные записи/ключи) ко всем нодам;
- ручное распределение контейнеров по узлам;
- необходимость самостоятельно настраивать балансировку нагрузки;
- ручной мониторинг и ручные перезапуски.

Со временем такая самодельная автоматизация часто превращается в трудно поддерживаемое решение.

### 1.3) Ключевое понятие Swarm – Service

В Swarm вместо управления «одним контейнером» используется **сервис**:

> **Service = образ + настройки + желаемое число реплик**

Вы задаете: «нужно 3 экземпляра приложения», а Swarm сам распределяет реплики по нодам.

Команда уровня идеи:

- `docker service create --replicas 3 ... <image>`

Важно: команды `docker service ...` выполняются **на manager**, а не на воркерах.

## 2) Kubernetes: что он дает поверх Docker-логики

Kubernetes – оркестратор промышленного уровня. Концептуально он похож на Swarm (кластер, распределение нагрузки, реплики), но функциональность шире:

- запуск и управление множеством реплик приложения;
- ручное и автоматическое масштабирование;
- **rolling updates** (постепенное обновление);
- **rollback** (откат на предыдущую версию);
- **canary / A-B** сценарии (обновление части инстансов);
- подключение сетей/хранилищ через плагины;
- встроенные механизмы аутентификации и авторизации.

Управление осуществляется через CLI **kubectl**.

### 2.1) Важная связь «Kubernetes и Docker»

Kubernetes использует **container runtime** – компонент, который реально запускает контейнеры на ноде.

Корректная модель:

- контейнеры остаются OCI-совместимыми (образы те же);
- runtime может быть **containerd** или **CRI-O**;
- поддержка Docker Engine как runtime в Kubernetes была убрана (dockershim removed), начиная с **v1.24**.

### 2.2) Архитектура Kubernetes «на пальцах»

Кластер Kubernetes – набор нод (машин/VM):

- **Worker node** – узлы, где запускается нагрузка (контейнеры/поды).
- **Control plane** – управляющая часть (ранее часто говорили «master»): следит за состоянием и оркестрирует.

Если в системе всего одна нода, ее падение означает падение приложения. Кластерная архитектура позволяет держать нагрузку на нескольких нодах и переносить ее при сбоях.

### 2.3) Базовые компоненты 

В Kubernetes обычно выделяются следующие компоненты:

- **API Server** – «входная дверь» в кластер: `kubectl` и панели управления общаются с ним.
- **Scheduler** – определяет, на какую ноду разместить новый workload.
- **Controllers** – «автопилот»: следит, чтобы текущее состояние совпадало с желаемым (создает/пересоздает/масштабирует).
- **kubelet** – агент на каждой ноде: обеспечивает запуск и здоровье контейнеров/подов.
- **etcd** – распределенное key-value хранилище состояния кластера.
- **Container Runtime** – запускает контейнеры (часто containerd/CRI-O).

### 2.4) Команды kubectl для старта

Минимальный набор команд:

- `kubectl cluster-info` – информация о кластере;
- `kubectl get nodes` – список нод;
- `kubectl get pods` – список подов;
- `kubectl apply -f <yaml>` – применить манифест (стандартный способ деплоя).

## 3) «Каким бывает Kubernetes»: варианты развертывания (актуально для 2026)

### 3.1) On-prem vs Cloud

- **On-prem**: вы самостоятельно обеспечиваете железо, сеть, ОС, кластер и интеграции.
- **Cloud**: часть слоев можно делегировать облачному провайдеру.

### 3.2) IaaS vs PaaS

- **IaaS** – «инфраструктура как сервис»: VM/сеть/диски/бэкапы. ОС и Kubernetes чаще всего в вашей зоне ответственности.
- **PaaS** – «платформа как сервис»: например, Managed Kubernetes, managed DB и др.

Чем выше уровень сервиса, тем меньше рутины на вашей стороне, но обычно выше стоимость.

### 3.3) Shared Responsibility Model (кто отвечает, если все упало)

Коротко:

> провайдер отвечает «за облако», пользователь отвечает «за то, что в облаке».

- В IaaS провайдер отвечает за работоспособность виртуальной инфраструктуры, но ОС/настройки внутри VM – ваша ответственность.
- В Managed Kubernetes провайдер обычно отвечает за control plane и часть инфраструктуры, но приложения/манифесты/секреты/права доступа – ваша зона.

### 3.4) Два практических подхода в облаке

#### (A) Self-managed Kubernetes на VM
Вы выбираете VM/ОС/инструменты, обслуживаете ноды и Kubernetes, выполняете обновления.

- Плюсы: гибкость, иногда дешевле по стоимости облачных ресурсов.
- Минусы: требуются компетенции и постоянное сопровождение.

#### (B) Managed Kubernetes
Кластер создается через консоль/CLI, провайдер обслуживает control plane и часто часть жизненного цикла нод.

- Плюсы: быстрый старт, меньше рутины, проще для новичков.
- Минусы: ограничения по кастомизации, иногда меньше доступа к нодам, стоимость.

## Где бесплатно потестить Kubernetes для лабораторной (2026)

### Вариант 1 – браузерные окружения (рекомендуется для аудиторных работ)
- **Killercoda Kubernetes Playgrounds** – Kubernetes в браузере без установки.

Плюсы: быстро стартовать, удобно проверять лабораторные «при преподавателе».

Примечание: Play-with-K8s исторически существовал, но его доступность может быть нестабильной, поэтому предпочтительнее использовать Killercoda.

### Вариант 2 – локально на ноутбуке (бесплатно)
Для практики дома/в аудитории можно использовать:

- `minikube`
- `kind`
- `k3d`

Это распространенные способы поднять локальный Kubernetes для обучения и разработки.

### Вариант 3 – Managed Kubernetes в облаке на кредитах
Для сценариев как в проде можно использовать trial/кредиты облаков (условия зависят от провайдера и типа аккаунта), но для учебного курса чаще подходят браузерные playground или локальные варианты (kind/minikube).
