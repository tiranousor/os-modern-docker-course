# Лекция: ConfigMap, probes и requests/limits в Kubernetes

## Зачем нужна эта тема

На прошлой практике мы уже развертывали приложение в Kubernetes, масштабировали его и обновляли версии.
Но в реальной эксплуатации этого недостаточно.

Обычно сразу возникают три практических вопроса:

1. **Куда вынести конфигурацию**, чтобы не пересобирать образ при каждом изменении?
2. **Как Kubernetes понимает**, что приложение уже готово принимать трафик или, наоборот, зависло?
3. **Как ограничить ресурсы**, чтобы один pod'ик не съел всю ноду и чтобы scheduler понимал, куда pod вообще можно поставить?

Именно это решают:

- **ConfigMap**
- **liveness/readiness probes**
- **requests/limits**

---

## 1. ConfigMap: внешняя конфигурация для приложения

### Что это такое

**ConfigMap** — это объект Kubernetes для хранения **несекретной конфигурации**.
Смысл: не монтировать настройки внутрь образа, а передавать их pod’ику снаружи.

Идея простая:

- образ приложения один и тот же;
- окружения разные (`dev`, `test`, `prod`);
- меняется не образ, а конфигурация.

То есть:

> **image = код приложения**  
> **ConfigMap = параметры его запуска / файлы конфигурации**

---

### Что обычно кладут в ConfigMap

Чаще всего туда выносят:

- текстовые настройки приложения;
- значения переменных окружения;
- `.properties`, `.conf`, `.yaml` файлы;
- HTML/шаблоны для простых демонстраций;
- адреса сервисов, флаги, режимы запуска.

Не стоит путать:

- **ConfigMap** — для обычной конфигурации;
- **Secret** — для чувствительных данных (пароли, токены, ключи).

---

### Как pod может использовать ConfigMap

Есть два основных способа.

#### 1) Как переменные окружения

Пример идеи:

- в ConfigMap хранится `APP_MODE=prod`
- pod получает это как env variable

Подходит, когда приложению удобно читать конфиг из `ENV`.

#### 2) Как смонтированные файлы

Пример идеи:

- в ConfigMap лежит `index.html`
- pod монтирует его в контейнер как файл

Подходит, когда приложение ожидает именно файл конфигурации.

---

### Почему это важно

Без ConfigMap часто получается плохой сценарий:

1. поменяли строку конфигурации;
2. пересобрали Docker image;
3. перепушили его;
4. сделали rollout.

Для простого изменения текста приветствия, порта или флага это избыточно.

С ConfigMap правильная модель такая:

- приложение остается тем же;
- конфигурация меняется отдельно;
- rollout и сопровождение становятся проще.

---

## 2. Probes: как Kubernetes проверяет состояние приложения

Очень частая ошибка новичка:

> «Container Running» ≠ «приложение уже готово и работает правильно»

Контейнер может быть запущен, но:

- приложение еще не поднялось;
- приложение зависло;
- веб-сервер запустился, но backend внутри не готов;
- сервис отвечает ошибками.

Поэтому Kubernetes использует **probes**.

---

### 2.1. Readiness probe

**Readiness probe** отвечает на вопрос:

> **Можно ли уже отправлять трафик в этот pod?**

Если readiness успешна:

- pod считается **Ready**;
- Service может направлять в него запросы.

Если readiness неуспешна:

- контейнер продолжает жить;
- pod помечается как **NotReady**;
- Service **не должен** направлять в него трафик.

Это очень важно во время старта приложения и во время временных проблем.

---

### 2.2. Liveness probe

**Liveness probe** отвечает на вопрос:

> **Живое ли приложение, или оно зависло так, что его надо перезапустить?**

Если liveness долго не проходит:

- kubelet считает контейнер сломанным;
- контейнер перезапускается.

То есть смысл liveness:

- не просто «готов/не готов»,
- а именно **нужен ли restart**.

---

### 2.3. Startup probe

Иногда приложение стартует долго.
Если сразу включить обычные liveness/readiness, они могут начать «бить» слишком рано.

Для таких случаев существует **startup probe**:

- пока startup probe не прошла,
- liveness/readiness фактически ждут.

В нашей базовой практике можно обойтись readiness + liveness, но про startup probe важно знать.

---

### Какие бывают проверки

Самые частые варианты:

- **HTTP probe** — Kubernetes делает HTTP-запрос (`/health`, `/ready`, `/`)
- **TCP probe** — проверяет, открывается ли порт
- **exec probe** — запускает команду внутри контейнера

Для учебной практики чаще всего удобно брать **HTTP GET**.

## 3. Requests и limits: ресурсы контейнера

Контейнер по умолчанию не должен жить по принципу «беру сколько хочу».

В Kubernetes есть два важных понятия:

- **request**
- **limit**

---

### 3.1. Request

**Request** — это минимальный объем ресурса, который pod просит у кластера.

Например:

- `requests.cpu: 100m`
- `requests.memory: 128Mi`

Смысл:

- scheduler учитывает requests;
- pod можно поставить только на такую ноду, где хватает ресурсов под эти requests.

То есть request — это **заявка на размещение**.

---

### 3.2. Limit

**Limit** — это верхняя граница использования ресурса контейнером.

Например:

- `limits.cpu: 500m`
- `limits.memory: 256Mi`

Смысл:

- контейнеру нельзя бесконтрольно расти;
- кластер получает более предсказуемое поведение.

---

### Как это работает на практике

#### CPU

Если контейнер упирается в `cpu limit`, происходит **throttling**:
ядро ограничивает доступ контейнера к CPU.

Иными словами:

- контейнер не обязательно падает,
- но начинает «тормозить», потому что ему режут CPU.

#### Memory

С памятью поведение жестче:

- контейнер может временно расти;
- при реальном давлении по памяти он может получить **OOM kill**.

Поэтому превышение memory limit опаснее на практике, чем упор в CPU limit.

---

### Почему pod может застрять в Pending

Если request слишком большой, scheduler не найдет подходящую ноду.

Тогда pod:

- будет создан как объект,
- но останется в состоянии **Pending**,
- а в событиях будет видно, что ресурсов не хватает.


---

## 4. Логика эксплуатации: как это все работает вместе

Нормальная продакшн-схема выглядит так:

1. Есть **Deployment**
2. Внутри контейнера:
   - настроены **readiness/liveness**
   - заданы **requests/limits**
3. Конфигурация вынесена в **ConfigMap**
4. Service отправляет трафик только в **Ready** pod’ы

Именно это делает deployment более «живым»:

- конфиг меняется отдельно;
- трафик идет только в готовые экземпляры;
- зависшие контейнеры перезапускаются;
- ресурсы не улетают в бесконечность.

---

## 5. Минимальный пример структуры YAML

Идея Deployment с этой темой обычно выглядит так:

- `metadata`
- `spec.replicas`
- `selector.matchLabels`
- `template.metadata.labels`
- `template.spec.containers`
  - `image`
  - `ports`
  - `envFrom` или `volumeMounts`
  - `readinessProbe`
  - `livenessProbe`
  - `resources.requests`
  - `resources.limits`

Дополнительно рядом обычно есть:

- `ConfigMap`
- `Service`
